## Features

- **Access Point Discovery** ‚ÄĒ Passive beacon frame sniffing with live table output. Extracts BSSID, SSID, channel, encryption, PMF (MFPC/MFPR), WPS status, DTIM period, group cipher, AKM suite, WPA1 downgrade detection, beacon interval, RRM (802.11k), BSS Transition (802.11v), and TSF uptime.
- **Connected Client Enumeration** ‚ÄĒ Identifies associated clients by inspecting data frames, EAPOL (4-way handshake), association responses, deauth, and disassoc frames. Tracks live Connected / Disconnected status per client.
- **Vulnerability Assessment** ‚ÄĒ Sends directed Probe Requests to a target AP and parses the Probe Response for a deep security profile: PMF state, group cipher, AKM suite, WPA1 downgrade risk, RRM/BSS-Transition support, and full WPS TLV breakdown (version, state, method, UUID-E, device info, setup-lock status).
- **PNL Extractor** ‚ÄĒ Passively captures directed Probe Requests from nearby devices to reconstruct their Preferred Network List (PNL) ‚ÄĒ the set of SSIDs a device has previously joined and will auto-associate with.
- **OUI Vendor Lookup** ‚ÄĒ Resolves AP and client MAC prefixes to vendor names using the IEEE OUI database.
- **Channel Hopper** ‚ÄĒ Threaded channel hopper supporting 2.4 GHz, 5 GHz, or interleaved dual-band sweeps.
- **Auto Dependency Installer** ‚ÄĒ Detects and installs missing Python dependencies via APT automatically.
- **Monitor Mode Management** ‚ÄĒ Switches interface to/from monitor mode via NL80211 (Netlink), with `airmon-ng` as a secondary enforcement fallback for quirky drivers.
- **Clean Shutdown** ‚ÄĒ Restores NetworkManager, wpa_supplicant, and the interface to managed mode on exit.

## How It Works

### Client Detection

Client association is determined through a layered signal approach rather than trusting association responses alone (which can report success even on a wrong password):

1. **Data frames** (`type=2`, `ToDS=1`) ‚ÄĒ client transmitting to AP proves active association.
2. **EAPOL frames** ‚ÄĒ 4-way handshake traffic confirms key exchange is in progress.
3. **Association Response** (`subtype=1`) ‚ÄĒ AP approval signal used as a secondary indicator.
4. **Deauth / Disassoc frames** (`subtype=0x0C` / `0x0A`) ‚ÄĒ marks client as `Disconnected`.
5. **Subsequent data / EAPOL after deauth** ‚ÄĒ automatically flips status back to `Connected`.

### Vulnerability Assessment

The assessment is triggered by injecting a crafted directed Probe Request at the target AP (with randomised source MAC, correct Supported Rates, Extended Rates, and HT Capabilities IEs) and then sniffing for the unicast Probe Response the AP sends back. The Probe Response carries the same Information Elements as a Beacon, so the same RSN / Vendor IE parser is reused. Fields extracted include:

- PMF state (MFPC / MFPR bits from RSN Capabilities)
- Group cipher and AKM suite type from the RSN IE
- WPA1 Vendor IE presence (downgrade attack surface)
- RRM (Tag 70, bit 0) and BSS Transition (Tag 127, bit 19)
- Full WPS TLV blob (Tag 221, OUI `00:50:F2:04`): version, configured state, PIN/PBC method, Selected Registrar flag, Setup Locked flag, RF bands, manufacturer, model, serial, device name, UUID-E, and primary device type

### PNL Extraction

Modern 802.11 clients periodically broadcast **directed Probe Requests** ‚ÄĒ management frames (type=0, subtype=4) carrying a specific SSID in the SSID IE (Tag 0) ‚ÄĒ to check whether a previously joined network is within range. These frames passively leak the device's **Preferred Network List (PNL)**: every SSID the device has connected to in the past and will auto-associate with if it hears a matching beacon.

Detection pipeline:

1. **Frame filter** ‚ÄĒ only management frames with subtype `0x04` are processed; everything else is dropped immediately.
2. **Wildcard suppression** ‚ÄĒ probes with an empty SSID IE (Tag 0, length 0) are discarded. These are undirected broadcast scans that reveal nothing about the PNL.
3. **Unicast source enforcement** ‚ÄĒ the LSB of the first octet of `addr2` is checked; multicast / broadcast sources are dropped as malformed.
4. **Deduplication by `(src_mac, ssid)` pair** ‚ÄĒ state is keyed on this tuple via `probe_index` (a dict), giving O(1) lookups. A device probing for multiple SSIDs produces one row per SSID.
5. **In-place update** ‚ÄĒ repeat probes to the same pair increment `Count` and refresh `Last Seen` without adding a new row.

**Security relevance:**

- **Privacy leak** ‚ÄĒ reveals SSIDs of networks a device has previously joined (home, office, hotel, etc.).
- **Evil Twin setup** ‚ÄĒ an attacker can respond to a directed probe with a rogue AP advertising the exact requested SSID, triggering automatic client association.
- **MAC randomisation note** ‚ÄĒ iOS 14+, Android 10+, and Windows 10+ randomise the source MAC per probe session, limiting persistent device tracking. The SSID content itself is still leaked regardless.
